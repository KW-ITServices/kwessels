Networking Theory (CompTIA Network+ aligned)

OSI Model — quick reference

  • Layer 7 — Application: HTTP, SMTP, FTP, DNS.
  • Layer 6 — Presentation: Encoding, encryption (TLS).
  • Layer 5 — Session: Sessions, sockets.
  • Layer 4 — Transport: TCP (reliable), UDP (fast).
  • Layer 3 — Network: IP addressing, routing (IPv4/IPv6).
  • Layer 2 — Data Link: MAC, Ethernet switching, VLANs, STP.
  • Layer 1 — Physical: Cables, fibres, electrical/optical signals.

Interview tip: When troubleshooting, identify the failing layer first — many ISP/home issues land in layers 1–3.

TCP/IP Stack — mapping & essentials

Application → Transport → Internet → Link. Know the difference between TCP and UDP, and be able to explain TCP 3-way handshake and how NAT affects IP mapping for home/business routers.

TCP handshake: SYN → SYN/ACK → ACK

Interview tip: Explain how port forwarding works and why NAT is used for IPv4 scarcity.

Addressing & Subnetting

Understand IPv4 classes, CIDR notation, subnet masks, and the method to calculate network, broadcast, and host ranges. Know common subnets (/24 = 255.255.255.0, 254 hosts).

Example: 192.168.1.0/24 → hosts 192.168.1.1 - 192.168.1.254

Interview tip: Practice a few quick subnet math problems aloud (e.g., split /24 into /26s).

Routing & Switching Basics

  • Switching: MAC learning, VLANs, STP prevents loops.
  • Routing: Static vs dynamic (OSPF, BGP basics).
  • NAT: Source NAT for outbound translation; port forwarding for inbound services.

Interview tip: Be ready to explain OSPF areas or BGP at a high level (why BGP is used between ASes).

Key technologies & protocols

  • DHCP, DNS, ARP, ICMP (ping), SNMP (monitoring), NTP.
  • 802.11 standards (a/b/g/n/ac/ax) — difference in bands, MIMO, channels.
  • VPN types — IPsec (site-to-site), SSL/TLS VPN (remote access).
  • Quality of Service (QoS) basics — prioritizing voice/critical traffic.

Topologies & Network Types

Network Topologies (what they are & when to use)

  • Star: Central switch/router — common in LANs. Pros: easy to isolate faults.
  • Bus: Legacy, shared medium — rare now.
  • Ring: Token ring historically; modern use in some carrier rings.
  • Mesh: Highly redundant — used in WAN backbones or wireless mesh.
  • Hybrid: Combination: practical for large enterprises.

Network Types

  • LAN: Local area network (office/home).
  • WAN: Wide area network — ISP/backbone links.
  • MAN: Metropolitan (city-wide mesh/fibre).
  • PAN: Personal (Bluetooth, USB tethering).
  • CAN: Campus area network (multiple buildings).

Fibre network fundamentals (intro)

Understand single-mode vs multimode fibre, attenuation, optical power budgets, and PON (Passive Optical Network) vs point-to-point fibre. PON is common for FTTH with splitters; ONT at customer premise converts optical to Ethernet.

Fibre & ISP Operations (PON, ONT, Installer Playbook)

PON architecture

PON uses an OLT (Optical Line Terminal) at the provider, splitters in the field, and ONTs at customer sites. Understand optical power budgets, reflective losses at connectors, and the importance of testing with an OTDR or power meter during install.

ONT vs Router

ONT converts light → Ethernet (Layer 1/2). Router handles WAN routing, NAT, DHCP, firewall. In troubleshooting, always isolate: test direct from ONT to laptop to eliminate router config issues.

Installer checklist (practical)

1) Confirm service activation & provisioning in OSS.
2) Check optical power (dBm) at termination point.
3) Verify fiber connector cleanliness (inspect probe).
4) Connect ONT, verify LOS/LOS alarm status.
5) Test Ethernet link to router; verify WAN IP/DHCP or PPPoE.
6) Perform speed test & document results.
7) Update ticket with test logs and photos; handover to customer.

Troubleshooting Library — Methods & Steps

CompTIA 7-Step Troubleshooting + ITIL overlay

  1. Identify the problem: Gather symptoms, scope, timestamps.
  2. Duplicate the problem: Attempt to reproduce.
  3. Establish a theory: Map to OSI layers and likely causes.
  4. Test the theory: Use ping, traceroute, direct connect tests.
  5. Plan & implement solution: Apply the smallest invasive fix first.
  6. Verify full system operation: Confirm with speed tests, logs.
  7. Document & close: Update incident/problem record, RCA if required.

Combine with ITIL: record communication, escalation, and change control if the fix requires system change.

Common toolset & commands

  • ping — connectivity & RTT
  • tracert / traceroute — path & hop latency
  • nslookup / dig — DNS resolution
  • ipconfig / ifconfig — IP config
  • netstat — open connections
  • Optical tester: OTDR, power meter for fibre

Example troubleshooting flows

Case A — "No internet but ONT light OK": 1) Direct connect laptop to ONT → check link; 2) Verify WAN IP assigned; 3) Check default gateway; 4) Ping gateway and public DNS (8.8.8.8); 5) If WAN IP missing, check ISP provisioning & OLT logs.

Case B — "Intermittent Wi-Fi": 1) Check client RSSI and interference (2.4GHz crowded); 2) Test wired speeds; 3) Run Wi-Fi heatmap / channel scan; 4) Recommend channel change, relocate AP, or deploy mesh nodes.

Cue Cards — Question & Answer with Troubleshooting Steps

Cue Card: "User can't log into domain"

Q
Answer summary: Check account status, network connectivity, DNS, time sync, and DC availability.

Troubleshooting steps:
  1. Verify user credentials & check if account is locked/disabled.
  2. Ask user to try on a different workstation to isolate endpoint vs account.
  3. Confirm the workstation is joined to the domain.
  4. Run nltest /dsgetdc:domainname or ping <DC> to test DC reachability.
  5. Validate DNS resolution: nslookup domaincontroller.
  6. Check event viewer on DC for 4625/4771 events (authentication failure codes).
  7. If needed, reset password or unlock account and test again; document actions and escalate if domain replication issues are suspected.

Cue Card: "Email not syncing (Microsoft 365)"

Q
Answer summary: Confirm license/provisioning, connectivity, and client config.

Troubleshooting steps:
  1. Verify user license & mailbox existence in Azure AD / Exchange Online.
  2. Check Outlook connectivity status (Work Offline mode, Cached Exchange Mode).
  3. Test mailbox via Outlook Web Access (browser) to determine server vs client issue.
  4. Run Test-Mailflow or use Microsoft Support & Recovery Assistant.
  5. If client profile corrupt, recreate profile or clear local OST file.
  6. Check network (firewall blocking ports) and DNS records (Autodiscover).
  7. Document steps & escalate to Microsoft support if tenant-side issues persist.

Cue Card: "Intermittent connectivity (fibre customer)"

Q
Answer summary: Isolate physical layer (fibre/ONT), validate router, and check for upstream errors.

Troubleshooting steps:
  1. Check ONT LEDs: LOS (loss of signal) indicates fibre issue.
  2. Use power meter or check field OTDR logs if available to inspect optical levels.
  3. Connect laptop directly to ONT to bypass router; test continuity and speed.
  4. Collect router logs for WAN disconnects; check DHCP/PPPoE status.
  5. Check for scheduled maintenance or upstream congestion via ISP status.
  6. If physical fault suspected, dispatch field tech with splice/connector inspection checklist.

Cue Card: "Slow Wi-Fi in one room"

Q
Answer summary: Evaluate signal strength, interference, and channel utilization; recommend relocation or mesh extension.

Troubleshooting steps:
  1. Check RSSI on client and AP (target RSSI <-65 dBm for good signal).
  2. Scan channels to find interference (neighbouring APs, microwaves).
  3. Test speeds wired vs wireless to confirm local wireless problem.
  4. Try changing channel or band (5GHz often less congested but shorter range).
  5. If coverage issue, recommend repositioning AP or adding a mesh node/EXT.

Cue Card: "Service degraded (slow but up)"

Q
Answer summary: Identify if degradation is WAN or LAN, find heavy flows, and apply temporary QoS/limits while resolving root cause.

Troubleshooting steps:
  1. Run speed tests from multiple endpoints and compare to package baseline.
  2. Check for bulk transfers, backups, or malware on hosts causing saturation.
  3. Monitor interface counters on router for errors, drops, CRCs (physical issues).
  4. Check ISP side for packet loss via traceroute and MTR.
  5. Apply rate limits or QoS class for business-critical apps while remediation is scheduled.

Scenario Playbooks — Timeboxed answers & actions

Scenario: "Enterprise customer — total outage" (2–5 minute opening)

Interview approach (what to say): "I would triage impact, confirm scope, isolate the layer, and keep stakeholders informed. Immediate remediation is focused on service restoration; post-incident we’d run RCA to prevent recurrence."

Playbook (practical steps):
  1. Confirm impact with customer: services affected & business risk.
  2. Check ONT/OLT, power, and fibre alarms (LOS/LOP).
  3. Test direct connectivity from ONT; if WAN is down, request upstream status and escalate to NOC.
  4. Open incident ticket, assign owner, and provide 15-minute updates until restored.
  5. After restoration, raise problem ticket for RCA; propose permanent fix & change plan.

Scenario: "Multiple customers report duplex mismatch symptoms"

Interview approach: Explain duplex mismatch and how you'd detect (errors, high collisions historically), and correct config mismatches or replace faulty NICs/switch ports.

Playbook:
  1. Collect interface stats: errors, late collisions, CRCs.
  2. Check switch & endpoint port settings (auto vs forced). Prefer auto/auto but match where required.
  3. Validate cable standard (Cat5e/6 for gigabit) and safe SFP modules for fibre.
  4. Replace cable/port and monitor for persistent errors; document remediation.
